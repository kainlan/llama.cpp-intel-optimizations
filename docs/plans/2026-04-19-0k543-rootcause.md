# llama.cpp-0k543 — C3 caller migration: raw-pointer escapes held by mem_handle / CHUNK_LEASE

**Status**: Writeup complete; migration in progress.

**Branch**: `feature/sycl-coalescing`
**Predecessors**:
- `vtf7f` T1-T3 (228de918a, 24dc137b2) — mem_handle lease refcount + `get_host_ptr(out_lease=)` + `cpu_mul_mat` weight lease
- `dyhdl` C1+C2 (4ab7f57bc, fc343063c) — chunk-level refcount + `from_chunk_ptr` factory + `resolve_slow` auto-pin
- `g0jrj` (747013179) — smart-pointer gate on MoE routing-subgraph CPU dispatch
- `a7l5w` (dca7494d6) — probe instrumentation + root cause analysis

**User directive (verbatim, non-negotiable)**:
> "No fallbacks, don't disable features, don't do work arounds, or take easy
> ways, always the correct approach"

**Architectural principle (from dyhdl)**: `mem_handle` IS the smart pointer.
Do not add parallel types. Extend the existing API (`from_chunk_ptr`,
`from_weight_lease`) at caller sites.

## 1. Scope

vtf7f+dyhdl shipped the lifetime primitives (`leased_entry_`,
`CHUNK_LEASE`, `from_chunk_ptr`, auto-pin in `resolve_slow`). C3 is the
remaining explicit migration for raw-pointer escape sites that still
hold a raw `void *` across a boundary where the backing can be
evicted / freed (cross-thread submit, host_task capture, async future,
graph-scheduler op ordering).

Sites NOT in scope (covered elsewhere):
- `cpu_mul_mat` src0 in the legacy-sync path — already leased via
  `src0_lease` (vtf7f T3, commit 24dc137b2).
- `resolve_slow` weight returns — auto-pin landed in dyhdl C2.
- `alloc_registry::lookup` for diagnostic / classification (`query_location`,
  `[KV-REMAP]` log, `a7l5w-probe`) — read-only, no pointer held for later
  use.

## 2. Inventory of raw-pointer escape sites

### Category A — `cpu_dispatch_lookup_host_ptr` callers

| File:line | Context | Risk | Protection today |
|-----------|---------|------|-------------------|
| cpu-dispatch.cpp:2496 | `get_host_ptr` fallback when unified-cache miss → returns registry mmap pointer | LOW — registry owns: (a) raw mmap ptr (process-lifetime) or (b) `aligned_alloc` buffer (registry-lifetime). Neither sits in a SYCL arena chunk, so no chunk lease would attach. | None, but also no race window: mmap and aligned_alloc lifetimes span inference. |
| cpu-dispatch.cpp:3955 | `cpu_mul_mat` async-mode src0 preference for persistent host copy | **Prior to C3, an async `host_task` captures the raw ptr by value.** When the registry holds `aligned_alloc` in CPU-offload mode, the pointer is stable for the registry's lifetime → low risk in practice. In the non-offload async path the captured pointer is the `set_tensor`-era raw ptr (mmap). | Async capture keeps pointer valid via value-copy lifetime of the registry entry. No lease, but path is documented to prefer the registry specifically because other sources are not lease-pinned. |

**Analysis**: The registry backs raw pointers that are either
process-lifetime (mmap) or registry-lifetime (`aligned_alloc`). Neither
sits in a `pinned_chunk_pool` chunk nor the VRAM arena — `from_chunk_ptr`
would fall through to DIRECT (no lease attached), which is the correct
semantics: a DIRECT handle documents "no arena protection needed".

**Migration**: Wrap the registry return in a `mem_handle::from_chunk_ptr`
at both callers. For non-arena pointers (the common case here) the
handle is a DIRECT no-op; for the uncommon case where a caller did
register a pointer that happens to land inside a pinned-pool chunk
(defence-in-depth only — not a currently known registrant), the handle
correctly acquires a chunk lease.

Consolidation: `g_host_ptr_map` is NOT redundant with alloc_registry.
The registry keys are raw pointers; the map keys are tensor names. They
serve disjoint lookup patterns. Collapsing would require name-indexing
the registry, a much larger surface than C3.

### Category B — `ggml_sycl_resolve_expert_ptr` callers (MoE expert raw pointers)

`ggml_sycl_resolve_expert_ptr` (ggml-sycl.cpp:2439) returns a raw
`const void *` from the unified cache's `lookup_expert` — no lease.
When the result is stored into a `cpu_expert_task::weight_host` and the
task is moved into a std::future or CpuExpertPool batch, the raw
pointer is held across an async completion boundary. If the expert
entry is evicted or its host chunk sycl::free'd during task execution,
the kernel dereferences a dangling pointer.

Call sites:

| File:line | Sync / async | Downstream use |
|-----------|--------------|----------------|
| ggml-sycl.cpp:34377 (moe_fusion `first_tasks`) | **ASYNC** — `cpu_pool.submit_batch(std::move(submit_tasks))` or `std::async` | `cpu_expert_mul_mat_batched` reads `task.weight_host` on worker thread |
| ggml-sycl.cpp:34596 (moe_fusion `down_tasks`) | ASYNC (same pool pattern around line 34610+) | Same |
| ggml-sycl.cpp:35203 (CPU-TG fast path primary tasks) | SYNC — `ggml_sycl_cpu_expert_mul_mat_batched(tasks_ptr, n_tasks)` within scope (line 35528) | Raw pointer used within the same scope; eviction is locked out by caller stack frame. LOW risk but still unguarded against a concurrent evictor on another thread (eviction grabs rw_mutex_ exclusive; this path runs without the cache lock). |
| ggml-sycl.cpp:35496 (CPU fallback for secondary GPU experts) | SYNC (same batched dispatch path at 35528) | Same |
| ggml-sycl.cpp:36475 (batch-dispatch `result.future = cpu_pool.submit_batch(std::move(batch_tasks))`) | **ASYNC** | Same |
| ggml-sycl.cpp:37475 (pending scatter `g_pending_scatter.future = cpu_pool.submit_batch(...)`) | **ASYNC** | Same |
| ggml-sycl.cpp:37872 (dispatch_host_expert_rows → `cpu_pool.submit_batch(...).get()`) | SYNC blocking — `.get()` is called in scope | The future is awaited before scope exits; raw ptr is stable for the pool's execution. MEDIUM risk — still survives across worker-thread queue-grab latency. |

**Async sites are the high-priority targets.** The pattern is:

```cpp
auto resolved = ggml_sycl_resolve_expert_ptr(src0, ctx.device, eid);
task.weight_host = resolved.ptr;  // raw pointer
...
cpu_pool.submit_batch(std::move(tasks));  // async boundary
```

Between `submit_batch` and the worker's `task.weight_host` dereference,
the cache entry can be evicted and its host chunk sycl::free'd.

**Migration**: `expert_resolve_result` grows a companion `mem_handle`
that holds a chunk lease on `resolved.ptr` for the life of the
enclosing task. `cpu_expert_task` grows a `mem_handle weight_lease`
field. The handle is moved with the task into the async batch so the
lease outlives the worker's read.

Additionally, when `lookup_expert` returns a HOST_PINNED entry, we can
acquire an entry-level lease via the cache API (the entry is an
`unified_cache_entry` just like DENSE_WEIGHT). For C3 we use
`from_chunk_ptr(resolved.ptr, device)` as the unified path — it
handles both VRAM- and host-arena-resident experts transparently, and
falls through to DIRECT for mmap-backed non-arena experts (which are
process-lifetime stable).

### Category C — `la.ptr + offset_within_layer` KV remap

Site: ggml-sycl.cpp:14618.

```cpp
void * old_data = tensor->data;
tensor->data    = static_cast<char *>(la.ptr) + offset_within_layer;
```

`la.ptr` is a per-layer KV arena allocation (either VRAM via
`unified_alloc` or host-pinned via unified cache). The assigned
`tensor->data` lives for the tensor's entire lifetime in the ggml graph
(context-lifetime, not per-op). The KV arena is context-lifetime —
freed at session end, not mid-inference.

**Analysis**: The reassignment is scope-durable through the buffer
context's destructor; `ctx->layer_allocs` owns `la.ptr` for the backend
buffer's lifetime. There is no mid-inference free path on this
pointer. Adding a per-tensor chunk lease would either:
1. Leak the lease forever (tensor never calls a "done" handler in ggml
   → 5s assert would fire at shutdown), OR
2. Require a new hook in the buffer destructor to release.

**Correct approach**: Attach a `mem_handle` (CHUNK_LEASE) to the
`ggml_tensor_extra_gpu` for this tensor at remap time; release in the
backend buffer's free path. That's a NEW ownership hook (extra owns a
mem_handle) — broader scope than C3 targets.

**For C3**: Acquire a one-shot context-lifetime chunk lease on `la.ptr`
at `layer_allocs` build time (once per layer, not per tensor),
released at context destruction. That gate prevents mid-inference
munmap of the KV arena chunk regardless of how many tensor->data
values point into it.

NB: currently, `la.ptr` comes from `unified_alloc` (VRAM) or a host
arena zone (host-pinned). When the source is host-pinned, the chunk IS
in the `pinned_chunk_pool`, so the 5s assert in `~pinned_chunk_pool`
would fire if context teardown races with lease release. We order
release BEFORE destroying `layer_allocs`.

### Category D — DnnlGemmWrapper scratchpad

`DnnlGemmWrapper::gemm / row_gemm / woq_gemm_q4_0` are GPU-side DNNL
primitives (device USM A/B/C pointers, device scratchpad). The
scratchpad is acquired from `ctx.pool()` (VRAM arena) per-call and
released on return from the submit. There is no host-side raw-pointer
handout across an async boundary — the primitive's queue submission
owns the scratchpad for its execution lifetime.

**No migration needed.** The scratchpad is VRAM-arena-allocated; if we
ever needed to protect it, we'd wrap it in a CHUNK_LEASE. Currently
it is bounded to the submit call and released immediately after, so
the chunk cannot be munmap'd mid-use (the arena chunks are
context-lifetime anyway).

## 3. Design notes

### `g_host_ptr_map` is NOT redundant with alloc_registry

- `alloc_registry` indexes by raw pointer (for lookup from ptr → info).
- `g_host_ptr_map` indexes by tensor name (for lookup from name → raw
  ptr during CPU dispatch when the tensor's GPU-side data has been
  staged device-only).

Collapsing would require name-indexing the registry. That is a
separate architectural change, OUT of scope for 0k543.

### Why `from_chunk_ptr` is the right API everywhere

- For cache-managed cache entries (DENSE_WEIGHT, MoE experts), the
  entry's `in_use_count` (vtf7f) and the underlying chunk's `lease_count`
  (dyhdl) are the two protection layers. `from_chunk_ptr` acquires the
  chunk lease. The entry lease is orthogonal: the existing `get_view`
  path gives chunk-only protection; `acquire_weight_lease` gives both.

- For MoE experts specifically, `acquire_weight_lease` takes a
  `unified_cache_key`, not an `ggml_sycl_cache_id` — the per-expert
  lookups use `lookup_expert` which returns a raw `weight_entry *`.
  Extending the cache API to return a leased `weight_entry` for
  `lookup_expert` is a bigger change. `from_chunk_ptr` lets C3 close
  the chunk-level hole without that extension.

- For non-arena pointers (mmap, external malloc), `from_chunk_ptr`
  degrades to DIRECT (no lease). This is correct — those pointers
  have lifetimes outside our management.

### Lease scope — per-task, not per-graph

Each `cpu_expert_task` carries its own `mem_handle` so the lease scope
exactly matches the async worker's read window. When the task is
moved into a pool batch (`std::move(submit_tasks)`), the mem_handle's
move constructor transfers the lease ownership (no refcount delta);
when the worker destroys the task, the lease releases.

This avoids the "held forever" trap: no thread_local, no static — the
lifetime is exactly the task vector's lifetime inside the worker.

### Performance

`from_chunk_ptr(ptr, device)` path:
1. `get_unified_cache_for_device(device)` — O(1) per-device array lookup.
2. `arena_acquire_chunk_lease(ptr)` — linear scan of `arena_chunks_`
   (max 2). O(1).
3. If miss: `host_acquire_chunk_lease(ptr)` — linear scan of host
   `chunks_`. Typically 2-6 chunks for 7B-20B; up to ~30 for 120B.
   O(chunks).

Under 20B the CPU expert dispatch path enters `from_chunk_ptr` once per
expert selected per layer — at most 4-8 per layer × 24 layers = ~100-200
calls per token. Per-call cost: <100 ns (cache array index + ~5 chunk
pointer compares + atomic fetch_add). Net impact: <20 us per token on a
model whose TG is ~75 ms/token. Negligible.

Per-call lease dtor: one `host_release_chunk_lease` or
`arena_release_chunk_lease` — same cost profile.

## 4. Regression risk + mitigations

| Risk | Mitigation |
|------|------------|
| Lease held forever in a stashed thread_local | All C3 leases live in per-task `mem_handle` fields that travel with the task into worker threads. Task destruction = lease release. |
| Per-dereference atomic on hot path | No. Lease is acquired at task-build time and released at task-destruction time. The worker's inner per-row SIMD reads pay NO atomic cost. |
| WEIGHT handle semantics changed | No. `from_chunk_ptr` is a distinct factory; existing WEIGHT paths unchanged. |
| `from_chunk_ptr` degrades mmap pointers to DIRECT | Correct: mmap pointers are process-lifetime, no chunk to refcount. |
| 5s assert fires at shutdown if a lease outlives the pool | Task vector destruction happens before pool destruction (pool joins all workers in its destructor; std::async future blocks scope exit). Verified by ordering. |

## 5. Gate-preservation reasoning

- **Mistral gates**: Mistral is dense, no MoE. The migrated sites
  (CPU expert dispatch) are not on the Mistral path. The cpu_mul_mat
  async registry-lookup path is also off-path for the canonical
  canary. Perf gates (PP ≥ 1700, TG ≥ 81) should be unchanged.
- **20B bench**: MoE path exercised. Chunk-lease acquire/release runs
  per expert per layer. Cost budget <20 us per token; TG budget is
  ~70 ms. Sub-percent overhead.
- **20B completion -n 128**: The lighthouse gate. If the async MoE
  routing still races, we've either missed a site or the race is at
  a layer below `cpu_expert_task`. Run A7L5W canary to confirm:
  `GGML_SYCL_A7L5W_INSTRUMENT=1` expects **zero probe aborts**.
- **120B bench**: Same cost profile, larger expert volume. Negligible.

## 6. Commit plan

1. **Commit 1** — extend `expert_resolve_result` with a `mem_handle
   lease` field and have `ggml_sycl_resolve_expert_ptr` acquire the
   chunk lease before returning. Downstream callers destructure the
   result and move the lease into their `cpu_expert_task`.
2. **Commit 2** — add `mem_handle weight_lease` to `cpu_expert_task`;
   migrate all 7 resolve-expert-ptr call sites to move their lease
   into the task.
3. **Commit 3** — migrate `cpu_mul_mat` async-mode registry lookup to
   acquire `from_chunk_ptr` and value-capture the handle into the
   host_task lambda.
4. **Commit 4** — KV `layer_allocs` context-lifetime chunk lease.
   Acquire at layer allocation, release at context destruction.

Each commit is buildable and self-contained. Commits 1-2 are the
highest crash-coverage work; commit 3 is the documented a7l5w async
write-capture hole; commit 4 is defensive depth for KV remap.
