### Dispatch flow for n=1

At `ggml-sycl.cpp:34087` the `ne12 == 1` branch enters the CPU-TG fast-path.
At `:34124` it checks `cpu_tg_val_fast != 0 && host_weights_fast && cpu_type_ok_fast`

That is **the test-visible** failure mode. It is a smaller problem than the 20B
bug, but it exposes the right code path: the CPU-TG fast-path that fires at
`ne12 == 1` when MoE weights are host-resident.

## 4. Second concrete observation — 20B takes a different CPU path

Under real 20B decode, allocations succeed (11+ GB host free), so the CPU-TG
alloc-failure symptom does not manifest. But `llama-completion` output is still
garbled. With `GGML_SYCL_DEBUG=1` enabled we see per-layer (all 24 MoE blocks):

```
[MoE] MMVQ layout=0 returned false
[MoE] Using expert cache for blk.N.ffn_up_exps.weight ...
[MoE] Falling back to host-side routing for type 39
```

MMVQ returns `false` at `mmvq.cpp:3745` — `effective != layout && !allow_aos_fallback`,
i.e. the base tensor's effective layout mismatches the requested SOA/COALESCED
route layout, and AOS fallback is not permitted. Control falls through to the
"host-side routing" branch at `ggml-sycl.cpp:35862`, which dispatches experts
via `ggml_sycl_cpu_expert_mul_mat_batched` using host mmap'd weight pointers.

This is **a separate code path from both** (a) the CPU-TG fast-path at line
34087 that fails in test-backend-ops, and (b) the GPU MMVQ path that works for
`n>=4` in test-backend-ops. The cpu-dispatch batched function IS the same one
both paths end up calling — but the wrapping logic differs.

Under `GGML_SYCL_VRAM_BUDGET_PCT=100` + `GGML_SYCL_CPU_EXPERT_TG=0` the dispatch
routes all 4 expert slices through `mmvq_moe_batched_dispatch` with
`route_layout=COALESCED` (since weights are registered host-resident via expert
cache) — this is what the `MUL_MAT_ID(n=1)` test can't exercise because its
fixture only has 4 experts and no cache staging.

## 5. What is NOT the bug (ruled out)

- **`e8m0_to_float_half`**: already hard-gated for `e<2`. Commit `87d9ff55a`
  fixed the denormal case. Current code matches the CPU reference.
- **`kvalues_mxfp4_unified` LUT**: identical across CUDA and SYCL.
- **`dequant_mxfp4_block_half8` vectorization** (Task A of the mxfp4-parity
  epic): `./build/bin/test-mxfp4-vector-dequant` passes `OK: 256×8 qs × e
  combinations match byte-exact (core + unaligned + aos)`.
- **`vec_dot_mxfp4_q8_1` signature** (March 7 hypothesis 62219): the SYCL
  variant does NOT take `kbx` but the caller pre-advances the pointer with
  `&x[ibx]`, which is arithmetically equivalent. Verified at `mmvq.cpp:708`
  and `:3533`-path for mul_mat_vec_q_id — pointer is offset correctly.
- **SIMD `simd_mxfp4_q8_0_16row` AVX-VNNI path**: inspection of the dequant
  (`cpu-dispatch.cpp:3202-3210`) matches the generic CPU reference
  (`ggml-cpu/quants.c:205-214`) — low-nibbles pair with Q8[0..15], high-nibbles
  with Q8[16..31], scale = q8.d * e8m0(xe) * 0.5 (the `*0.5` is embedded in
  `GGML_E8M0_TO_FP32_HALF`).

## 6. What IS broken — specifically

The 8 failing test-backend-ops entries all have `n=1`, the exact shape that
fires the CPU-TG fast-path. The reason they produce bad numbers is
**the 1 KB `managed_host_pinned_buffer::ensure` alloc fails** under the
test-backend-ops constraints (test uses 370 MB arena budget after pre-probe
reserves 512 MB slack headroom + 256 MB scratch + 256 MB oneDNN).

Under real 20B inference the 11+ GB host arena has no allocation pressure for
the 1 KB staging buffer, so CPU-TG does complete its compute. But the **20B
garbled-output symptom stems from a different CPU-dispatch site**: the
host-side-routing fallback at `ggml-sycl.cpp:35862`, specifically the
`dispatch_cpu_compute` lambda at `:36436`.

Concrete evidence that the dispatch-level plumbing is the culprit (not the
MXFP4 math kernel itself):

- Same SIMD kernel (`simd_mxfp4_q8_0_16row`) is called in both `MUL_MAT` tests
  (pass) and `MUL_MAT_ID` tests (fail at n=1), via the same vec_dot path.
- Disabling CPU-TG completely (`GGML_SYCL_CPU_EXPERT_TG=0`) makes all 69
  MUL_MAT_ID tests pass.
- 20B output is still garbled with `GGML_SYCL_CPU_EXPERT_TG=0` because 20B's
  MMVQ-false path (host-side routing) goes through `dispatch_cpu_compute`, NOT
  CPU-TG, and that path has its own wrapper lambda that builds the
  `batch_tasks` vector and submits to `g_cpu_expert_pools`.

The **shared target** of both failures — `ggml_sycl_cpu_expert_mul_mat_batched`
(cpu-dispatch.cpp:810) — would produce correct results if given correct inputs.
The wrapper lambdas are where inputs get corrupted.

## 7. Why I did NOT implement a fix this session

Four reasons, each independent:

1. **Parallel agent on `cpu-dispatch.cpp` / `ggml-sycl.cpp`**: the mission brief
   explicitly says this file is off-limits to me while bead `0k543` (C3 lease
   migration) is in flight. The uncommitted `git diff` on those files at the
   start of this session confirmed the other agent has pending lines inside the
   exact lambdas (`get_host_ptr`, `cpu_mul_mat`, `dispatch_cpu_compute`) that
   I would need to modify. A stomp would block both beads.

2. **The real 20B bug is not a simple arithmetic error in the dequant kernel**.
   Every piece of MXFP4 math (e8m0 scale, LUT lookup, block indexing,
   vectorization) I audited matches the CPU reference byte-for-byte. The bug
   is in how tasks are built and/or how their outputs are scattered back to
   device memory. That class of bug is exactly what the `0k543` lease-migration
   work is addressing.

3. **test-backend-ops does not reproduce the exact 20B failure mode**. The
   test-visible failures are all alloc-starvation artefacts of the test fixture,
   not the compute bug we care about for Gate 4. A CPU-TG fix (make the 1 KB
   alloc work) would turn test OK but leave 20B garbled because 20B doesn't hit
   CPU-TG — it hits the host-side-routing fallback.

4. **"No fallbacks, don't disable features" per user directive.** Workarounds
   like `GGML_SYCL_CPU_EXPERT_TG=0` improve coherence but only partially
   (some runs produce the right numeric sequence, others don't). That's a flag
   dependency, not a fix. Committing that would violate the directive.

This is the highest-leverage fix path — directly addresses the 20B garbled

### 12.5 Cross-backend handoff mechanics

ggml scheduler at ggml-backend.cpp:1435-1446 has a `skip_copy=true` branch
for MoE-weight inputs that reach a different backend — no staging copy is
created; the downstream backend reads `src->data` directly.

For host-planned MoE weights (SYCL_Host buffer, `is_host=true`), the CPU
backend takes over via `supports_op=false`. The CPU backend then reads
`src0->data + cur_a * nb02` from the SYCL_Host pinned memory.  Architecturally
this should be correct — the SYCL_Host buffer type reports `is_host=true`
(via CPU buffer type's interface at ggml-sycl.cpp:17973), so
`ggml_backend_sched_buffer_supported(sched, src, CPU_backend)` returns
true; no copy is interposed, and CPU memcpy reads pinned bytes.

### 12.6 Non-determinism → likely race on the CROSS-BACKEND intermediate tensors, NOT weights

Weights are written once at model load and never mutated. The only cross-backend
intermediate that gets recomputed per token is the F32 activation between
splits. Specifically:

1. `ffn_moe_logits-N` (MUL_MAT gate router, GPU)
2. `argsort / top-k / softmax` → ids (GPU)
3. `ffn_moe_gate-N`   (MUL_MAT_ID, CPU — routed there because gate_exps is host-planned)
4. `ffn_moe_gate_biased-N` (ADD_ID, CPU)
5. `ffn_moe_up-N`     (MUL_MAT_ID, CPU)
6. `ffn_moe_up_biased-N` (ADD_ID, CPU)
7. `ffn_moe_weighted-N` (GLU, CPU)
8. `ffn_moe_down-N`   (MUL_MAT_ID — GPU for layers 0-8, CPU for 9-23)
9. `ffn_moe_down_biased-N` (ADD_ID, CPU)

The cross-backend boundary tensors that must be correctly synchronized:
- ids (GPU → CPU)
- ffn_moe_weighted (CPU → GPU for layers 0-8; stays on CPU for 9-23)
- ffn_moe_down output (GPU for layers 0-8 → CPU for biased; all-CPU otherwise)

Each D2H / H2D boundary is a synchronization obligation on the scheduler.
A missed sync or a stale `input_cpy` reused across splits would produce
tokens that differ by exactly the bytes the unsynchronized backend wrote.
Non-determinism across identical inputs is consistent with that failure
mode.

between GPU and CPU backends straddles data that is not being serialized
correctly.

### 12.11 Revised root-cause locus

The bug is INSIDE SYCL's cpu-dispatch path (`ggml_sycl_cpu_expert_mul_mat_batched`
and/or `dispatch_cpu_compute` / `moe_fusion`), or in its interaction with
the SYCL graph replay / expert prefetch / pool lease infrastructure.  The
n04bq §10 candidate B ("audit dispatch_cpu_compute lambda") was correct
— but the test-backend-ops CPU path (standalone ggml-cpu backend) does
NOT hit the same code path as SYCL's internal cpu-dispatch, so the 618/618
eval pass on ggml-cpu does NOT exonerate SYCL's cpu-dispatch.

Specifically, SYCL cpu-dispatch path for host-resident MoE weights:
1. Pre-quantizes F32 activation to Q8_0 in pinned host buffer
2. Calls `simd_mxfp4_q8_0_16row` or similar SIMD kernels with
   `host_weight_ptr + expert_offset` as src0
3. Writes output to pinned host, then H2D back to VRAM for downstream GPU op

Failure modes that match "non-deterministic garbled output":
- Stale `host_weight_ptr` (resolver returns different pointer per run for
  same key — lease discipline not fully enforced)
- Pre-quantization buffer race (thread-local T1 buffer aliased across experts
  in the same batch)
- H2D write to output buffer overlaps with downstream GPU kernel's read
  (missing event dependency)

Next session should add targeted probes inside cpu-dispatch.cpp that log
`host_weight_ptr`, `act_host_ptr`, and output offsets per expert, across
3 consecutive TG steps. If any pointer differs run-to-run for the same
(tensor, expert_id, token_id), that's the race site.  This investigation
is out of scope for this session's file-scope constraint (cpu-dispatch.cpp
is owned by the 0k543 parallel agent per mission brief).

### 13.3 Phase 1 — race site pinpointed

```
$ diff <(grep L144I run1.log) <(grep L144I run2.log) | head -2
< cf/dst_exit tok=-1 tensor=__fattn__-1 idx=36 hash=0x3f54de5ed9217a08 max_abs=78.72
> cf/dst_exit tok=-1 tensor=__fattn__-1 idx=36 hash=0x3d1cef22c7d722bd max_abs=12.65
```

idx=0..35 are **bit-identical** across runs. Including:
- token embeddings
- all layer-0 attention (SWA): Qcur-0, Kcur-0, Vcur-0, `__fattn__-0`
- layer-0 post-attention norm, residual, ffn_up, ffn_down, and bias
- layer-1 RMS norm input and output
- layer-1 Qcur (3 permutations), Kcur (3), Vcur (2)
- the "(copy)" op right before fattn-1

**idx=36 `__fattn__-1`** produces different output given bit-identical
inputs. The XMX flash-attention kernel is **not deterministic**.

Further evidence:
- `__fattn__-0` (layer 0, SWA): identical across runs (hash `0x464d23...`)
- `__fattn__-1` (layer 1, full attn): diverges
- `__fattn__-2` (layer 2, SWA): **diverges** (hash differs, max_abs
  2415 vs 1.473e+07 — wildly different because downstream compounding)
- `__fattn__-3..N`: all diverge

So **layer 0 SWA is deterministic; layer 1+ full attention is not**. Once
fattn-1 produces divergent output, every downstream layer amplifies it
and the final token distribution changes, producing different tokens.

### 13.4 Characterizing the race — NaN poisoning

Past layer 3, run 1 shows:
```
idx=7 blk.3.ffn_down_exps.weight src1 n=230400 nan=23040 min_abs=1.336e-16 max_abs=28.15
```

11520 NaN = exactly 4 (experts-per-token) × 2880 (head_dim) at layer 1.
By layer 3 both runs have 23040 NaN = 8 × 2880 elements. Non-finite
values propagate through downstream MUL_MAT_ID ops and produce NaN
outputs that the router then feeds back into the activation stream.

The position of NaN varies across runs — run 1 has 2880 NaN at idx=2
(layer 1 ffn_up src1); run 2 has 0 NaN at the same position but 2880
NaN by idx=6 (layer 3 ffn_up src1). The race converts tiny float-level
variations into NaN cascades.

### 13.5 Why layer 0 (SWA) is deterministic and layer 1 (full attn) is not

The layers differ in KV cache size:
- SWA layers (0, 2, 4, ..., 22): kv=1.5 MB (768 tokens sliding window)
- Full layers (1, 3, 5, ..., 23): kv=256 MB (131072 tokens max)

Both go through `ggml_sycl_flash_attn_ext` → `ggml_sycl_flash_attn_ext_dispatch_ncols`
→ `launch_fattn_xmx_f16` (XMX path on Arc B580).

The XMX kernel (`fattn-xmx-f16.hpp`) processes KV in `XMX_BATCH_KV` tiles
with double-buffered K^T loads and per-subgroup joint_matrix MADs.
For SWA's tiny 768-position KV, the reduction fits in one or two
tiles. For full attention's 131072-position KV (even if only the
first few positions have real data for a prompt of 20 tokens), the
reduction spans many more tiles, and accumulation behavior diverges
across runs.

Probable specific culprit (from inspection at `fattn-xmx-f16.hpp:978-999`):

The kernel distributes work as `(d_tile, k_tile)` pairs across subgroups,
with partial results written to SLM and reduced in PHASE 4.5
(`NEED_K_REDUCTION`). The comment at line 986 is ominous:
```
// Results are atomically accumulated (or we use deterministic assignment)
```

Either (a) the "deterministic assignment" branch has an off-by-one that
leaves some tiles uninitialized (NaN from SLM trash) or (b) concurrent
reads of `SV_acc` happen before `group_barrier`. Need to audit PHASE 3
(Q@K^T), PHASE 4 (S@V), PHASE 4.5 reduction, and all `group_barrier`
placements for read-after-write without a prior barrier.

### 13.6 Scope of fix — XMX flash attention kernel

**This is not a cpu-dispatch bug.** cpu-dispatch, moe_fusion,
dispatch_cpu_compute, and all the candidate hot spots from §8 and §11
are NOT in the failure path. Predecessor agents were chasing the wrong
file.

The fix is inside `ggml/src/ggml-sycl/fattn-xmx-f16.hpp`, specifically in
the `flash_attn_xmx_f16_kernel` template. Likely fixes:

1. **Add a `group_barrier` before PHASE 4.5 K-reduction reads** so that
   all work items have finished writing their partial sums to SLM before
   any thread starts summing them.
2. **Initialize the unused `SV_acc` tiles** (indices `work_id >=
   TOTAL_WORK_ITEMS` and `d_idx >= D`) explicitly to 0 so the K-reduction
   doesn't sum stale SLM.
3. Audit the `tile_KT` double-buffer hand-off: the "END-of-loop barrier
   removed — next iteration's Q@K^T barrier provides sync" comment at
   line 1097 is a hazard. If a previous iteration's K-load finishes
   AFTER the current iteration's Q@K^T starts reading `tile_KT_cur`, we
   read half-written K data.
