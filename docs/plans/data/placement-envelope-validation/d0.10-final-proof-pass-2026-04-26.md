# D0.10 Final Proof Pass - 2026-04-26

## Question

Is there any remaining unproven architectural work before executing the Unified
Memory Placement plan?

## Findings

### Dense and SWA+MoE planning

Backed for the tested local families:

- Mistral 7B dense/full-attention: D0.1 mini-vs-real reserve and D0.2 PP/TG
  union pass.
- GPT-OSS 20B active SWA+MoE: D0.1 and D0.2 pass; runtime evidence records
  `sliding_window=128`, `n_swa=128`, and split SWA/non-SWA KV caches.

This is enough to start Track A for dense and active SWA+MoE, but not enough to
claim every architecture.

### State-space / SSM

Not backed. Static evidence:

- `src/models/graph-context-mamba.cpp` and `src/models/plamo2.cpp` emit
  `ggml_ssm_scan`.
- `ggml/include/ggml.h` defines `GGML_OP_SSM_SCAN`.
- CPU, CUDA, Metal, and Vulkan have visible `GGML_OP_SSM_SCAN` support.
- `ggml/src/ggml-sycl` has no `GGML_OP_SSM_SCAN` support/routing hit.

Conclusion: state-space support requires both `llama.cpp-xwnkf` (SYCL
`SSM_SCAN` implementation or explicit CPU route) and `llama.cpp-3h5gm.4`
(fixture canary). Until then, state-space is excluded from close criteria.

### Multi-device transfer representation

Original C2 premise refuted. D0.3a synthetic two-GPU proof records stable
scheduler copy-edge tensors but no `GGML_OP_CPY` nodes:

```text
[D0.3a] synthetic run 1: backends=3 splits=1 copies=1 nodes=5 leafs=2 cpy=0 scheduler_copy_edges=2
[D0.3a] synthetic run 2: backends=3 splits=1 copies=1 nodes=5 leafs=2 cpy=0 scheduler_copy_edges=2
```

Stable transfer tensors:

- `SYCL1#d0_3a_input_a#0`
- `SYCL1#d0_3a_input_b#0`

Conclusion: C2 must plan scheduler transfer edges separately from executable
graph ops. Walking only `GGML_OP_CPY` nodes is wrong.

### Unified-cache ownership / smart handles

Not backed. Static and runtime evidence still shows direct allocation and raw
pointer compatibility paths. Representative current callsites:

- `ggml/src/ggml-sycl/common.hpp:3183`: graph-input staging uses
  `sycl::malloc_device()`.
- `ggml/src/ggml-sycl/cpy.cpp:729-740`: TP copy staging uses host/shared USM
  wrappers.
- `ggml/src/ggml-sycl/dense-scheduler.cpp:20-21`: dense scheduler slots use
  `ggml_sycl_malloc_device()`.
- `ggml/src/ggml-sycl/cpu-dispatch.cpp:287-289`: CPU offload host copy uses
  `aligned_alloc(64)`.

Runtime guard evidence already caught `kv_tier:migrate` using
`ggml_sycl_malloc_host(queue)` outside unified-cache ownership. Conclusion:
D0.7b-D0.7d remain required before claiming the smart-handle end state.

### CPU-dispatch-vs-streaming priority policy

Not backed. Existing D0.8 proof attempts failed because the streaming path hits
Level Zero OOM/device-loss and CPU-dispatch correctness/scratch sizing fails.
The priority benchmark must wait until Track E arena-overcommit blockers are
fixed and `test-sycl-cpu-dispatch` is repaired.

## Execution Readiness

The plan is executable if scoped as:

- Start with Track A/A3/A3a/A4/A7 for dense and active SWA+MoE.
- Use placement-envelope planning for model-scope max capacity, not per-user
  context at model load.
- Keep state-space, scheduler-transfer implementation, CPU-dispatch policy, and
  full mem-handle ownership as explicit open follow-ups.

The plan is not ready if the claim is "all architectures and all memory
ownership are fully proven." Those claims are still false.
