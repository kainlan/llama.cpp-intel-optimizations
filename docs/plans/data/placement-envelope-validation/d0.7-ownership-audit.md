# D0.7 Unified-Cache Ownership Audit

**Date:** 2026-04-25
**Epic:** `llama.cpp-3h5gm` Unified Memory Placement
**Status:** scoped, not closed

## Question

Can the epic currently claim that unified_cache owns all SYCL inference memory and that ggml SYCL hot paths no longer bypass smart handles?

## Finding

No. The direction is right, but the claim is not yet proven. Production SYCL still has direct allocation callsites and raw pointer access patterns that must be classified or moved before epic close.

This audit used:

```bash
rg --count-matches \
  "ggml_sycl_malloc_device\\(|ggml_sycl_malloc_host\\(|ggml_sycl_malloc_shared\\(|ggml_sycl_malloc_device_raw\\(|sycl::malloc_device|sycl::malloc_host|sycl::malloc_shared" \
  ggml/src/ggml-sycl -g '!**/tests/**' -g '!**/docs/**' -g '!**/dpct/CLAUDE.md'

rg --count-matches \
  "tensor->data|extra->data_device|extra->data_handle|data_device_ptr" \
  ggml/src/ggml-sycl -g '!**/tests/**' -g '!**/docs/**'
```

## Raw Pointer / Smart Handle Scan

The production scan found 273 references to `tensor->data`, `extra->data_device`, `extra->data_handle`, or `data_device_ptr`.

| File | Count | Initial classification |
|---|---:|---|
| `ggml/src/ggml-sycl/ggml-sycl.cpp` | 220 | Mixed. Contains resolver/buffer-boundary code, tensor load/set/get paths, KV remap, graph input staging, TP setup, logits, and some dispatch-adjacent direct reads. Needs detailed classification. |
| `ggml/src/ggml-sycl/common.hpp` | 27 | Resolver implementation / smart-handle bridge. Mostly allowed if it remains the only hot-path access boundary. |
| `ggml/src/ggml-sycl/common.cpp` | 10 | Tensor extra cleanup and buffer bookkeeping. Likely allowed boundary. |
| `ggml/src/ggml-sycl/set_rows.cpp` | 4 | Direct extra pointer diagnostics / dispatch. Needs classification. |
| `ggml/src/ggml-sycl/getrows.cpp` | 3 | Embedding/table pointer path. Needs classification. |
| `ggml/src/ggml-sycl/cpu-dispatch.cpp` | 3 | View-chain host pointer resolution. Likely allowed CPU-dispatch boundary if leases are held. |

Initial rule for closing D0.7:

- Resolver/buffer-boundary code may read `tensor->data` only to create or refresh a `mem_handle`, register a stable host source, calculate a view offset, or service public tensor get/set APIs.
- Kernel dispatch code should consume `mem_handle`/`data_device_ptr()`/placement-plan outputs, not raw `tensor->data`, except for explicit CPU-dispatch host-source paths with a lease or immutable host storage guarantee.
- Every remaining per-token or per-op allocation must be tied to a placement-plan arena zone, preallocated scratch/staging pool, or a named exemption.

## Plan Impact

D0.7 remains open. The audit is enough to prove the epic cannot claim full unified-cache ownership yet. It should now be implemented as a classification/cleanup pass, not left as vague risk.

Recommended bead handling:

- Keep `llama.cpp-9lkgd` open until every row above is classified.
- Add child beads only for callsites classified as bugs after the detailed pass; do not create one bead per grep hit.
- Treat `fattn_weight_stage`, `dense_scheduler_slot`, and `mmvq_*temp/compact` as likely first cleanup candidates because they are closest to inference hot paths.
- Implementation cleanup is tracked by `llama.cpp-3h5gm.3`; allocator allowlist enforcement by `llama.cpp-3h5gm.5`; handle-boundary cleanup by `llama.cpp-3h5gm.6`; and ggml/SYCL logical-storage proof by `llama.cpp-3h5gm.7`.

## Deeper End-State Audit — 2026-04-25

The end goal is stricter than "most allocations call a wrapper": unified_cache
must own active-path SYCL memory, and non-allocator code must receive handles or
leases, resolving raw pointers only at the final kernel or memcpy boundary.

### Architectural gaps

1. **Allocator ownership is not mechanically enforced.** There is a direct
   allocation guard (`GGML_SYCL_DIRECT_ALLOC_GUARD`), but the plan lacks a
   checked allowlist that fails CI or D3 when non-unified-cache files own SYCL
   memory.

2. **`mem_handle` is not yet the only ownership-facing API.**
   `ggml_tensor_extra_gpu` still stores `data_device[]`; `data_handle[]`
   resolves first and falls back to the raw pointer. That is a migration bridge,
   not the end state. Direct `data_device[]` access remains in cleanup/debug and
   TP worker paths.

3. **The ggml/SYCL boundary still depends on raw `tensor->data` and
   `get_base()+offset` semantics.** Some uses are unavoidable public
   `ggml_backend_buffer` boundary code today, but relocatable/segmented
   storage is only sound if non-contiguous backing is hidden behind backend
   logical storage or remapped into handles before op dispatch. Exposing
   segmented host storage as one flat base pointer is architecturally unsound.

4. **Direct allocation wrappers are still compatibility paths.**
   Calls such as `ggml_sycl_malloc_device()` can route through the cache when
   possible, but they still allow raw fallback allocation. End-state planned
   execution should request a typed `alloc_request` / lease and fail if the plan
   did not reserve capacity.

### Updated close rule

D0.7 cannot close until:

- A checked allocator allowlist exists and all non-allowlisted active-path
  allocation/free sites are removed or converted to unified-cache leases.
- `data_device[]` is private compatibility state, not read directly outside the
  resolver/cleanup boundary.
- Dispatch code resolves via `mem_handle` / `ggml_sycl_resolve()` / plan output
  and keeps any needed lease alive across async work.
- The plan audit (`D2`) proves no planned PP/TG path allocates outside
  unified-cache after warmup.

### Runtime guard evidence

A guarded runtime probe added concrete evidence that the static scan reflects
real execution paths. With `GGML_SYCL_DIRECT_ALLOC_GUARD=2`, a small Mistral
GPU run reached KV-clear failure handling and attempted a host migration through
a direct wrapper:

```text
[SYCL] direct allocation bypassed unified_cache:
api=ggml_sycl_malloc_host(queue) size=2.00 MB tag=kv_tier:migrate phase=unknown
GGML_SYCL_DIRECT_ALLOC_GUARD detected a direct allocation outside unified_cache
```

The same run also logged planned-vs-actual KV divergence:

```text
[KV-TIER] DIVERGENCE: layer 15 planned=host actual=device
...
[KV-TIER] 17/32 layers diverged from placement plan
```

The device-loss portion of that run happened under concurrent stress and should
not be treated as a new standalone driver finding. The allocation-guard hit is
still valid D0.7 evidence: `kv_tier:migrate` must be routed through
unified-cache leases or explicitly classified before ownership can be claimed.

## 2026-04-26 Additional Fact Check

The latest proof pass re-ran targeted static scans and confirmed the ownership
gap is still architectural, not just stale documentation:

```bash
rg -n "dpct::dpct_malloc|malloc_device|malloc_host|sycl::malloc|ggml_sycl_malloc|aligned_alloc" \
  ggml/src/ggml-sycl ggml/src/ggml-backend.cpp src/llama-context.cpp src/llama-kv-cache.cpp
```

Representative current findings:

- `ggml/src/ggml-sycl/common.hpp:3183` still has graph-input staging that
  allocates with `sycl::malloc_device()` directly.
- `ggml/src/ggml-sycl/cpy.cpp:729-740` still has TP copy staging that can call
  `ggml_sycl_host_malloc()` or `ggml_sycl_malloc_shared()`.
- `ggml/src/ggml-sycl/dense-scheduler.cpp:20-21` still allocates persistent
  `dense_scheduler_slot` buffers through `ggml_sycl_malloc_device()`.
- `ggml/src/ggml-sycl/cpu-dispatch.cpp:287-289` still owns an
  `aligned_alloc(64)` copy path for CPU offload host pointers.
- `ggml/src/ggml-sycl/unified-cache.cpp` still contains allowed raw allocator
  internals for arena reservation, but also fallback paths that must be
  guarded by the allocator allowlist and placement-envelope admission policy.

This does not change the implementation direction. It reinforces the split:
`llama.cpp-3h5gm` owns the placement plan and scheduler/arena integration, while
`llama.cpp-mubmt` / `llama.cpp-860ks` own the broader mem-handle API cleanup.
The Unified Memory Placement epic cannot claim "unified_cache owns all SYCL
memory" until D0.7b-D0.7d are complete.
