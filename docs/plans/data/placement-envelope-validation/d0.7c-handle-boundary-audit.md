Each reference is tagged with one of:

- **ALLOWLIST** — inside a resolver/boundary function (set/get/init_tensor
  callback, public buffer-type API, resolver primitive, weight registration,
  cleanup boundary). Raw read is intentional and remains.
- **ALLOWLIST_COMMENT** — comment-only mention of `->data` or
  `data_device[]`. Documentation, not a functional read.
- **ALLOWLIST_DEBUG** — `GGML_LOG_*` / `GGML_SYCL_DEBUG` / `fprintf(stderr,...)`
  diagnostic logging. Allowed under the debug exemption.
- **MIGRATE_RESOLVER** — sync dispatch path that should call
  `ggml_sycl_resolve_tensor_ptr(t, device)` instead. Bulk of the migration.
- **MIGRATE_LEASE** — async/host-view path that needs a `mem_handle` /
  `weight_lease` whose lifetime extends across the SYCL event chain.
- **BUG** — defect (e.g., dangling read, missing lease, dangling fallback).
  File a separate bead.
- **REVIEW** — heuristic could not classify; needs human eyes.
  Counted as remaining audit work in the per-line table below.

Classifier output is heuristic. ALLOWLIST_COMMENT and ALLOWLIST_DEBUG
are mechanically reliable; everything else needs human review during the
Commit 2/3 migration passes.

### Resolver-semantics contract

`ggml_sycl_resolve_tensor_ptr(tensor, device)` (in
`ggml/src/ggml-sycl/common.hpp`) returns the **device-resident
USM pointer** for `tensor` on `device`, with these specific
behaviors:

1. **Fast path**: if `tensor->extra->data_handle[device]` is
   populated and resolves successfully, returns that handle's `.ptr`.
   The handle is populated during weight preload (cache-id) or
   `set_data_device` (direct, e.g., for KV cache or compute
   activations).
2. **Non-weight slow path**: calls `ggml_sycl_get_data_ptr` →
   tries `extra->data_device_ptr(device)` → checks
   `alloc_registry`: if `tensor->data` is `DEVICE` or `HOST_PINNED`,
   populates the handle and returns `tensor->data` itself.
   Otherwise falls through to `ggml_sycl_get_data_ptr_slow`
   (view-src walk, mmap detection, staging copy).
3. **Weight slow path**: `ggml_sycl_get_layout_ptr_impl` performs
   layout-aware resolution (SOA/COALESCED/AOS via the unified cache).

For tensors backed by a SYCL device buffer-type, the resolver is
byte-equivalent to `tensor->data` once `data_device` has been set
by `init_tensor`. For tensors on **host buffers** (e.g.,
`SYCL_Host`), the resolver returns `tensor->data` if it's
`HOST_PINNED` USM; for non-pinned host memory (mmap'd, plain malloc),
the slow path may stage to device. **The resolver always returns
a pointer the GPU can reach — never the original host pointer for
non-pinned host memory.**

This contract has two consequences relevant to the inline review
below:

- **Per-buffer-type dispatcher branches** (e.g., `cpy_tensor_2d`)
  cannot call the resolver from inside an `if (is_X)` arm because
  the dispatcher itself is the source-of-truth for which buffer
  type the tensor has. The resolver is the abstraction *over* this
  dispatcher; calling it from within the implementation is a
  layering violation, even when byte-equivalent.
- **Host-destination consumers** (e.g., the `!dst_on_device` branch
  in `ggml_sycl_op_mul_mat`) need the actual host pointer for D2H
  staging memcpy. The resolver would return a device pointer, which
  would land the result in VRAM and silently break the consumer.
  This is a **strict correctness** requirement, not a stylistic one.

## Notes on the heuristic

The classifier is intentionally conservative: it tags references
`ALLOWLIST` only when the enclosing function name appears in an
explicit allowlist of resolver/boundary primitives. References inside
larger dispatch functions (e.g., `ggml_sycl_op_mul_mat`) get `REVIEW`
even if the actual usage might be allowlist-eligible (e.g., the
function reads `tensor->data` only to pass it to a resolver). Human
review is required.

`(none)` in the enclosing-function column means the classifier could
not identify the enclosing function — typically a lambda, closure,
anonymous namespace, or template instantiation. These are flagged
REVIEW automatically; manually identify the context during Commit 2/3
triage.

For Commits 2/3, the `REVIEW` lines are the work surface. Each line
should be inspected and re-tagged as one of the four real categories
(ALLOWLIST / MIGRATE_RESOLVER / MIGRATE_LEASE / BUG) before the
migration commit lands.
