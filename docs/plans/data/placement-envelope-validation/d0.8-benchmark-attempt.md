# D0.8 CPU Dispatch vs GPU Streaming Benchmark Attempt

**Date:** 2026-04-25  
**Epic:** `llama.cpp-3h5gm` Unified Memory Placement  
**Status:** blocked / unproven

## Question

Can the plan treat CPU dispatch for host-resident weights as benchmark-backed and delete GPU streaming fallback paths?

## Finding

No. The first available benchmark path did not produce performance data. It failed in the GPU streaming/compute path with Level Zero device loss.

## Commands

```bash
source /opt/intel/oneapi/setvars.sh --force

env -u ONEAPI_DEVICE_SELECTOR \
  LD_LIBRARY_PATH=build/bin:$LD_LIBRARY_PATH \
  SYCL_DEVICE_FILTER=level_zero:gpu \
  GGML_SYCL_MMVQ_BENCH=1 \
  GGML_SYCL_MMVQ_BENCH_MODE=both \
  GGML_SYCL_MMVQ_BENCH_ITERS=3 \
  GGML_SYCL_MMVQ_BENCH_WARMUP=1 \

  SYCL_DEVICE_FILTER=level_zero:gpu \
  GGML_SYCL_MMVQ_BENCH=1 \
```

## Results

`GGML_SYCL_MMVQ_BENCH_MODE=both` failed with:

```text
terminate called after throwing an instance of 'sycl::_V1::exception'
what(): level_zero backend failed with error: 20 (UR_RESULT_ERROR_DEVICE_LOST)
```

The backtrace showed failure during `sycl::queue::memcpy()` from the benchmark's GPU streaming path.

`GGML_SYCL_MMVQ_BENCH_MODE=mmq` failed with:

```text
level_zero backend failed with error: 20 (UR_RESULT_ERROR_DEVICE_LOST)
[SYCL-SYNC-DIAG] DEVICE_LOST on device 0: cache_used=0.0 MB, cache_budget=1.0 MB
```

The MMQ run also logged that the forced 1 MiB cache budget refused both single-chunk and two-chunk arena reservation, then fell back to per-entry allocation before device loss.

parallel/pathological setup:


Backtrace:

```text

main()
```

So D0.8 is blocked by correctness failures and CPU-dispatch scratch sizing, not
only by a generic segfault.

## Plan Impact

D0.8 remains open and should block C3 deletion of streaming controls. Current evidence supports a safer policy statement:

- CPU dispatch is the intended data-local path for host-resident weights.
- Streaming remains unproven and currently unstable under the available benchmark.
- C3 may remove shadow heuristics only after D0.8 either proves CPU dispatch across dense + MoE workloads or explicitly preserves a fallback/recovery path for cases where CPU dispatch is not validated.

## Follow-up Shape

The next D0.8 pass should:

- Run the kernel benchmark with a budget that does not force pathological arena refusal unless that is the scenario being tested.
- Include a dense model canary and a MoE canary, not only synthetic Q8_0 MMVQ/MMQ.
- Record correctness, tok/s or ms/op, PCIe traffic if available, and whether any path device-lost.
- Treat `UR_RESULT_ERROR_DEVICE_LOST` as a benchmark failure that blocks policy deletion, not as missing data.
- `llama.cpp-3h5gm.2` tracks repair of the CPU-dispatch and streaming benchmark
  proof harnesses before D0.8 is rerun.
