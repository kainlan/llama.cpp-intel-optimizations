# D0.9 Remaining Proof Audit

**Date:** 2026-04-25  
**Epic:** `llama.cpp-3h5gm` Unified Memory Placement  
**Status:** additional proof gaps found; D0.1 gap closed 2026-04-25

## Question

After D0.5-D0.8 and the deeper D0.7 ownership audit, is there any remaining
proof work before implementation starts?

## Finding

Yes. The end-state architecture is still coherent. One stale canary gap was
closed after this audit (D0.1), while state-space, multi-device CPY,
CPU-dispatch policy, and unified-cache-only ownership remain open.

## Evidence

### D0.1 canonical skeleton determinism was stale; now corrected

Initial audit finding: the checked-in `test-planner-canary-skeleton-determinism`
still wrote an `INCONCLUSIVE` result with `reason=blocked_on_m09zb` and exited
0. That was a valid placeholder before the patched compute-runtime fix, but it
was not a current proof artifact:

```bash
rg -n "intentionally SKELETAL|blocked_on_m09zb|TODO: when E1 lands|return 0" \
  tests/test-planner-canary-skeleton-determinism.cpp \
  docs/plans/data/planner-canaries/d0.1-skeleton-determinism.md \
  tests/data/planner-canaries/d0.1.json
```

Resolution on 2026-04-25: the canonical D0.1 canary was replaced with an
orchestrator over `test-mini-context-prototype`. It now runs the real-A /
real-B / mini reserve protocol on all available fixtures and writes PASS for
Mistral 7B dense plus GPT-OSS 20B active SWA+MoE.

### State-space is more than a missing fixture

The local model inventory still has no Mamba/RWKV/SSM GGUF, so D0.2 remains
unvalidated for recurrent/state-space model families. A static source audit
adds a stronger gap: Mamba/Mamba2 and Plamo2 graph builders emit
`ggml_ssm_scan`, but the SYCL backend has no `GGML_OP_SSM_SCAN` support entry.

```bash
rg -n "GGML_OP_SSM_SCAN|ggml_sycl.*ssm_scan|ssm_scan" ggml/src/ggml-sycl

rg -n "ggml_ssm_scan|GGML_OP_SSM_SCAN" \
  src/models ggml/src/ggml-cpu ggml/src/ggml-vulkan ggml/src/ggml-sycl
```

Observed result:

- SYCL search: no `SSM_SCAN` implementation or support entry.
- Graph builders: `src/models/graph-context-mamba.cpp` and
  `src/models/plamo2.cpp` call `ggml_ssm_scan`.
- CPU and Vulkan have `GGML_OP_SSM_SCAN` handling.

Implication: the plan may proceed for the locally proven families
(dense/full-attention and GPT-OSS active SWA+MoE), but it cannot claim
architecture-general state-space support until a fixture proves the graph shape
and the SYCL execution/routing story handles `SSM_SCAN`.

### Multi-device CPY-node visibility is refuted; scheduler copy edges are stable

The host can expose two SYCL devices when `GGML_SYCL_SPLIT_RATIO` is set:

```bash
source /opt/intel/oneapi/setvars.sh --force
env -u ONEAPI_DEVICE_SELECTOR \
  LD_LIBRARY_PATH=build/bin:$LD_LIBRARY_PATH \
  SYCL_DEVICE_FILTER=level_zero:gpu \
  GGML_SYCL_SPLIT_RATIO=50,50 \
  timeout 90 ./build/bin/llama-bench --list-devices
```

This reports `SYCL0` B580 and `SYCL1` B50. Without split env, the backend
intentionally hides secondary GPUs from the scheduler:

```text
[SYCL] Multi-GPU: exposing only device 0 to scheduler
(2 physical GPUs available internally for MoE/split)
```

Follow-up implementation on 2026-04-25 added a D0.3a multi-device mode to
`test-planner-canary-cpy-visibility` and a scheduler debug accessor for the
last split graph. The target builds, and a real run with
`D0_3_MULTIDEVICE=1 GGML_SYCL_SPLIT_RATIO=50,50 SYCL_DEVICE_FILTER=level_zero:gpu`
does expose both GPUs and enters multi-device layer placement. It still does
not answer the CPY question: the run timed out after 300 s during the first
GPU-offloaded context construction, before split-graph inspection could run.

Observed blocker evidence:

- direct host allocations through `ggml_sycl_malloc_host(queue)` for
  `kv_tier:layer_18` through `kv_tier:layer_31`;
- KV placement divergence for layers 15-17 (`planned=host actual=device`);
- graph inspection not reached, so no CPY count/name snapshot exists.

Follow-up proof on 2026-04-26 changed D0.3a to a synthetic scheduler graph so
it could isolate the scheduler representation from full llama context/KV
blockers:

```bash
source /opt/intel/oneapi/setvars.sh --force
D0_3_MULTIDEVICE=1 \
GGML_SYCL_SPLIT_RATIO=50,50 \
SYCL_DEVICE_FILTER=level_zero:gpu \
LD_LIBRARY_PATH=build/bin:$LD_LIBRARY_PATH \
timeout 120 ./build/bin/test-planner-canary-cpy-visibility
```

Result:

```text
[D0.3a] synthetic run 1: backends=3 splits=1 copies=1 nodes=5 leafs=2 cpy=0 scheduler_copy_edges=2
[D0.3a] synthetic run 2: backends=3 splits=1 copies=1 nodes=5 leafs=2 cpy=0 scheduler_copy_edges=2
```

The stable transfer tensors were `SYCL1#d0_3a_input_a#0` and
`SYCL1#d0_3a_input_b#0`. Therefore the old C2 premise is wrong: cross-backend
transfers are represented as scheduler split-input copy edges (`GGML_OP_NONE`
tensors), not `GGML_OP_CPY` graph nodes, in this path.

Implication: C2's op-id keying is validated for single-device scheduler order,
but multi-device transfer planning must consume scheduler copy-edge metadata
instead of walking only `GGML_OP_CPY` nodes.

### D0.8 has a concrete CPU-dispatch harness failure

The previous D0.8 note recorded `test-sycl-cpu-dispatch` as a segfault. A gdb
rerun shows the more precise failure:

```bash
source /opt/intel/oneapi/setvars.sh --force
env -u ONEAPI_DEVICE_SELECTOR \
  LD_LIBRARY_PATH=build/bin:$LD_LIBRARY_PATH \
  SYCL_DEVICE_FILTER=level_zero:gpu \
  timeout 45 gdb -batch -ex run -ex bt --args ./build/bin/test-sycl-cpu-dispatch
```

Result:

```text
test_int4_kernel_correctness ... FAIL (task 1 should be full precision, max_diff=52.630264)
test_adaptive_split ... FAIL (expected 3 full-precision matches, got 1)
test_non_q4_0_fallback ... FAIL (task 1: Q6_K should be full precision, max_diff=56.199848)
ggml/src/ggml-sycl/cpu-dispatch.cpp:462:
GGML_ASSERT(q_row_size <= g_cpu_dispatch_buffers.src1_q.size()) failed
```

Backtrace ends in `ggml_sycl_cpu_vec_dot_rows()` called by
`test-sycl-cpu-dispatch`. This proves D0.8 is blocked by correctness and buffer
sizing failures in the CPU-dispatch proof harness, not just by a flaky
benchmark executable.

### Runtime direct-allocation guard catches a real bypass

A guarded run with `GGML_SYCL_DIRECT_ALLOC_GUARD=2` reached the KV-clear
failure path and then tried to migrate a KV layer through a direct host
allocation:

```text
[SYCL] direct allocation bypassed unified_cache:
api=ggml_sycl_malloc_host(queue) size=2.00 MB tag=kv_tier:migrate phase=unknown
GGML_SYCL_DIRECT_ALLOC_GUARD detected a direct allocation outside unified_cache
```

The same run logged planned-vs-actual KV divergence:

```text
[KV-TIER] DIVERGENCE: layer 15 planned=host actual=device
...
[KV-TIER] 17/32 layers diverged from placement plan
```

The device-loss trigger in that run occurred under concurrent stress, so it
should not be treated as a new standalone driver finding. The allocation guard
hit is still useful: it confirms at runtime that D0.7's static direct-allocation
audit is not merely theoretical.

## Plan Impact

Additional upfront work should be tracked before the epic claims the remaining
proof gaps are closed:

- D0.1 replacement is complete; keep its new PASS artifact as the canonical
  A3a proof for tested dense/SWA+MoE families.
- Keep D0.3a open only to update/close the implementation contract: the proof
  now refutes CPY-node visibility and validates stable scheduler copy-edge
  naming.
- Treat state-space as two pieces: acquire/generate a fixture and add/prove the
  SYCL `SSM_SCAN` execution/routing path.
- Keep D0.8 blocked until CPU-dispatch correctness and scratch sizing pass, and
  until streaming-vs-CPU benchmark data exists for dense and MoE.
- Keep D0.7 cleanup/enforcement open; runtime guard evidence now shows at least
  `kv_tier:migrate` bypassing unified-cache ownership.

## Conclusion

The plan is not ready to execute as "proof complete." It is ready to execute
only if Track A starts with the known proof gaps explicitly excluded from its
claims: dense/SWA+MoE single-device planning and A3a mini-context sizing are
backed; state-space, scheduler-transfer implementation, CPU-dispatch-vs-
streaming policy, and unified-cache-only ownership are not yet backed.
