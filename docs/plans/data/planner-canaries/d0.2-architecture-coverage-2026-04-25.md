# D0.2 Architecture Coverage Check - SWA and State-Space

Date: 2026-04-25

## Question

Does the D0.2 "single-shape sizing suffices" proof still lack SWA and
state-space coverage?

## Evidence

Local GGUF inventory contains no obvious state-space or independent SWA
fixture:

```bash
find /Storage/GenAI/models -maxdepth 1 -type f \( -name '*.gguf' -o -name '*.GGUF' \) | sort |
  rg -i 'mamba|rwkv|jamba|state|ssm|recurrent|hybrid|gemma|mixtral|qwen|llama-3|llama3|sliding|swa'
```

The command produced no matches.

Mistral 7B Q4_0 is a dense/full-attention local baseline:


GPT-OSS 20B MXFP4 is not MoE-only. It has active sliding-window attention:


The canonical D0.2 canary was re-run on 2026-04-25 with both baseline models:

```bash

env -u ONEAPI_DEVICE_SELECTOR \
  LD_LIBRARY_PATH=build/bin:$LD_LIBRARY_PATH \
  ONEAPI_DEVICE_SELECTOR=opencl:cpu \
```

Result: PASS. The committed canonical evidence remains byte-identical:

- Mistral Q4_0: PP=13 ops, TG=13 ops, `PP_only=""`, `TG_only=""`.
- GPT-OSS 20B: PP=17 ops, TG=17 ops, `PP_only=""`, `TG_only=""`.

The codebase has state-space architecture support, so the missing fixture is
a real coverage gap rather than an irrelevant model family:

```text
src/llama-arch.h: LLM_ARCH_MAMBA, LLM_ARCH_MAMBA2, LLM_ARCH_RWKV6, LLM_ARCH_RWKV7
src/llama-arch.h: LLM_KV_SSM_*, LLM_TENSOR_SSM_*
ggml/src/ggml-cpu/ggml-cpu.c: GGML_OP_SSM_CONV, GGML_OP_SSM_SCAN, GGML_OP_RWKV_WKV6, GGML_OP_RWKV_WKV7
```

## Finding

SWA is now partially validated. The active GPT-OSS 20B alternating SWA + MoE
path produces identical PP/TG op sets, so the previous "SWA absent locally"
claim was too broad.

State-space remains unvalidated. No local Mamba/RWKV/SSM GGUF exists under
`/Storage/GenAI/models/`, and those architectures use distinct model/KV/op
surfaces. The plan should keep a state-space follow-up bead before claiming
universal architecture coverage.

Additional static audit: state-space is also not proven on the SYCL execution
path. `src/models/graph-context-mamba.cpp` and `src/models/plamo2.cpp` build
graphs using `ggml_ssm_scan`, while `ggml/src/ggml-sycl` has no
`GGML_OP_SSM_SCAN` support/dispatch entry. CPU and Vulkan do have
`GGML_OP_SSM_SCAN` handling. That means the state-space follow-up must prove
both graph coverage and SYCL execution/routing for `SSM_SCAN`; a fixture alone
is not enough.

## Impact

A3a can proceed for the locally proven families: Mistral dense/full-attention
and GPT-OSS active SWA + MoE. It should not claim state-space coverage until a
state-space fixture is acquired or generated, D0.2 is rerun against it, and the
SYCL `SSM_SCAN` support/routing gap is closed or explicitly routed to CPU.
