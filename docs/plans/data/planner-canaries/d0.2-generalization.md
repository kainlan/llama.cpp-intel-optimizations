# D0.2 Generalization — Multi-Model Op-Union Check

Extends the original D0.2 PASS (Mistral 7B Q4_0 + GPT-OSS 20B MXFP4) to
additional GGUF variants on the host machine. The canary runs PP-shape
(`ubatch=512`) and TG-shape (`ubatch=1`) `graph_reserve` + decode for
each model and compares the captured op sets.

## Op sets captured

**Mistral 7B (every quantization)** — identical 13-op set:
```
ADD, CPY, FLASH_ATTN_EXT, GET_ROWS, GLU, MUL, MUL_MAT,
PERMUTE, RESHAPE, RMS_NORM, ROPE, SET_ROWS, VIEW
```

**GPT-OSS 20B MXFP4** — 17-op set (Mistral set + 4 MoE-related):
```
ADD, ADD_ID, ARGSORT, CPY, FLASH_ATTN_EXT, GET_ROWS, GLU, MUL,
MUL_MAT, MUL_MAT_ID, PERMUTE, RESHAPE, RMS_NORM, ROPE, SET_ROWS,
SOFT_MAX, VIEW
```

GPT-OSS adds `ADD_ID`, `ARGSORT`, `MUL_MAT_ID`, `SOFT_MAX` over the dense
Mistral set — those are MoE routing ops (top-k expert selection plus the
batched-by-expert MUL_MAT_ID), not architecture-shape differences.

## Worker-side notes

- Every worker run terminated with `SIGABRT` from `ggml_vec_dot_f16`'s
  `!isnan(sumf) && !isinf(sumf)` assertion. This is the **expected**
  CPU-backend behavior on a BOS-only prompt (the FA softmax produces
  NaN, vec_dot trips); the canary's two-phase fork+exec design plus the
  signal handler captures the partial op set before the abort. Crash
  signal 6 is recorded in `*_decode_crash_sig` evidence keys.
- `ASK_CALLS` was 32 across all Mistral runs and 48 across the GPT-OSS
  20B run, matching the layer count × ops-per-layer pattern (32-layer
  Mistral vs 24-layer GPT-OSS with extra MoE per-layer ops).

## Architectural-coverage caveat

This system has **only two architectures locally**:
- Mistral 7B v0.1 (dense, Llama-style) — 11 quantization variants available, 5 tested (Q4_0 baseline + Q2_K, Q3_K_M, Q4_K_M, Q8_0)
- GPT-OSS in MoE variants (20B + 120B-3-part)

Note: Mistral 7B v0.1 was originally trained with sliding-window
attention but llama.cpp's runtime treats it as full attention;
v0.2/0.3 dropped SWA at training time. So the dense-Mistral baseline
does **not** by itself exercise an active SWA codepath, and the
generalization claim below is over dense + MoE specifically.

No Llama 3.x, Qwen 2.x, Gemma, Mixtral, or state-space models are
present at `/Storage/GenAI/models/`. The 5 new datapoints are
quantization variations of the dense baseline; per the planning
rubric this is "better than no data" (plan §Task 2 "Notes for
implementer") but does **not** rule out architecture-shape PP/TG
asymmetry on:
- sliding-window attention (Gemma 2 alternating SWA/full, Mixtral)
- state-space models (Mamba, RWKV)
- different attention layouts (multi-query, GQA-with-sinks)

GPT-OSS 120B (3-part split, MoE) was not tested in this session — its
60 GB CPU-only PP=512 decode is intractable on the canary's existing
compute path (~hours per shape). Re-test once an op-only path that
skips compute is added (out-of-scope for this canary version).

## Overall finding

**PASS — generalization holds across all tested variants.** The PP-ops ==
TG-ops invariant survives quantization-format change on the dense
baseline and is preserved on the MoE 20B baseline. Single-shape
`graph_reserve` continues to be a sufficient sizing pass for A3a within
the architecture coverage available on this host.

## Recommendation

A3a can size from a single shape on the architectures tested. Before
deploying broadly, re-run the canary on (a) a sliding-window-attention
model (Gemma 2 alternating SWA/full, Mixtral) and (b) a state-space
model (Mamba/RWKV) once such GGUFs are present locally. If either FAILs, A3a
acquires a per-architecture branch; if both PASS, the single-shape
sizing claim is confirmed across all production-relevant architectures
and the union strategy in D16 ("PP + TG graph union sizing") is
formally not required for the dense + MoE + state-space + SWA matrix.

## Reproducer

```bash
source /opt/intel/oneapi/setvars.sh --force
ninja -C build test-planner-canary-pp-tg-union

OUT=docs/plans/data/planner-canaries/d0.2-gen
mkdir -p "$OUT"
for variant in Q2_K Q3_K_M Q4_K_M Q8_0; do
    LD_LIBRARY_PATH="/Apps/llama.cpp/build/bin:$LD_LIBRARY_PATH" \
      MISTRAL_PATH="/Storage/GenAI/models/mistral-7b-v0.1.${variant}.gguf" \
      GPTOSS_PATH="" \
      ONEAPI_DEVICE_SELECTOR=opencl:cpu timeout 300 \
      ./build/bin/test-planner-canary-pp-tg-union
    # Lowercase, strip underscore so output keys match the committed snapshot
    # filenames (q2k.json, q3km.json, q4km.json, q8.json):
    key=$(echo "$variant" | tr 'A-Z_' 'a-z' | sed 's/^q//;s/^/q/')
    cp tests/data/planner-canaries/d0.2.json "$OUT/${key}.json"
done
```
