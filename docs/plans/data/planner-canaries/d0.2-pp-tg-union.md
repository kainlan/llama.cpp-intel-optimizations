## Bonus finding: in-process multi-context crash (independent repro of `llama.cpp-m09zb`)

The first few iterations of this canary used a single-process design
that created two `llama_context` instances from the same `llama_model`
back-to-back: a PP-shape context, collected ops, freed it, then a
TG-shape context on the same model. On this host (Intel Arc B580 via
`level_zero:0`, oneAPI 2025.3 + patched compute-runtime 26.09.37435.10)
the second `llama_new_context_with_model` silently exited in every
attempt:

- Stderr cut off cleanly between the HOST-ARENA zone-configure line
  and the first `llama_context: ...` line of the second context.
  Exit code 0, no abort, no SYCL error message.
- Even with `n_gpu_layers=0` the SYCL backend remained registered and
  its `unified_cache` was involved in the teardown of the first context;
  subsequent attempts to build the second context aborted inside
  `ggml_backend_sycl_synchronize` with `UR_RESULT_ERROR_DEVICE_LOST`
  (20) once the first context had already reserved the VRAM arena.
- Serialising the contexts (PP ctx created → decode → freed → only
  then TG ctx created) did not change the outcome: `unified_cache` state
  from the first context persisted into the second.

These observations independently match implementer-1's D0.1 repro of
`llama.cpp-m09zb` (staging_buffer_pool multi-context re-entry).
Two canaries independently hit the same failure class is strong signal
the fix must land as epic prerequisite E1 before multi-context support
can be claimed elsewhere in the design.

The canary as committed avoids the bug with a **process-isolation
workaround** (driver forks+execs itself once per shape), which is
stronger than in-process serialization and happens to produce complete
op-set data for the D0.2 question. See the `D0.2 Recovery` section of
`tests/test-planner-canary-pp-tg-union.cpp` comments for the rationale.
