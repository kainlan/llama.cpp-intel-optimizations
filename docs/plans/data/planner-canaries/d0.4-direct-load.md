# D0.4 — INCONCLUSIVE

**Summary**: Canary cannot complete today: runtime execution wedges in
`ggml_backend_sycl_buffer_set_tensor` (L0 event.wait hang) at the first
`ggml_backend_tensor_set` call.

**Recommendation**: blocked_on_m09zb — rerun after E1 lands

**Reason**: blocked_on_m09zb (L0 DirectSubmission non-flush)

## What this canary would test when unblocked

Direct mmap -> device tensor_set with byte-identical readback, in one
`ggml_backend_tensor_set` call. Measures `tensor_set_us` (one call including queue submit + sync). Proves (or
disproves) that A7's weight loader can write mmap'd GGUF bytes directly
into a pre-allocated device buffer at a chosen offset without any
host-pinned intermediate.

## Evidence

The canary source (`tests/test-planner-canary-direct-load.cpp`) builds
cleanly and uses only `ggml-backend` APIs — no `llama_model_load_from_file`,
no `llama_init_from_model`, no `unified_cache_direct_stage_weight`. It
therefore does NOT traverse the S1-PRELOAD staging path that was the
subject of the original m09zb RCA (Task 6). Running it nevertheless
wedges under `timeout 120 s` with `EXIT=124`.

### What this shows

The hang signature (sched_yield spin in `L0::EventImp::queryStatus`,
controller asleep in `pthread_cond_wait`) is **identical** to the
m09zb wedge inside `staging_buffer_pool::acquire` / S1-PRELOAD. But the
call chain is entirely different: `ggml_backend_sycl_buffer_set_tensor`
at `ggml-sycl.cpp:12685` does `queues_wait_and_throw()` + a single
`stream->memcpy(...).wait()`. No staging pool, no preload, no mutex
held across the wait. The wedge still fires.

So m09zb is not a "staging_buffer_pool" bug — it is an **L0
DirectSubmission non-flush** affecting *any* `event.wait()` post-init
under the current driver/compiler combination on Arc B580:

- oneAPI 2025.3
- libze_intel_gpu.so (xe driver stack)
- Intel Arc B580 on Ubuntu 26.04, kernel 7.0

Fix scope must widen accordingly: E1's acceptance should include
non-staging `event.wait()` callsites (D0.4 canary as the second marker,
in addition to the D0.1 canary).
